# 2026-06 面试问题复盘

来源：[面试进展](../05_进展跟踪/面试进展)

## 总体判断

这两次面试的问题并不偏怪，主要集中在五类：

- C 语言基础：`volatile`、`static`、指针大小、作用域、浮点数比较。
- Linux/驱动基础：I/O 处理、线程同步、驱动 class 注册。
- 调试能力：根据堆栈地址定位函数。
- 基础算法表达：链表排序、两个栈实现队列。
- 平台知识：ARM、ACPI/DSDT、设备描述方式。

复盘重点不是背更多零散知识，而是把每类问题回答成“定义 + 场景 + 坑点 + 项目关联”。

## 1. I/O 相关处理怎么回答

面试答法：

> I/O 可以从用户态接口、内核驱动接口和硬件访问三个层面理解。用户态常见接口是 `open/read/write/ioctl/mmap`，驱动侧一般通过 `file_operations` 暴露能力；如果涉及硬件寄存器，驱动会通过 `ioremap` 把 MMIO 物理地址映射到内核虚拟地址，再用 `ioread/iowrite` 访问。实际项目中还要考虑阻塞/非阻塞、超时、错误恢复、并发保护和日志定位。

可以补充：

- 阻塞 I/O：没有数据或资源时线程睡眠等待。
- 非阻塞 I/O：立即返回，常配合轮询或事件机制。
- `ioctl`：适合传递控制命令，比如配置设备、查询状态。
- `mmap`：适合把设备或文件映射到用户态，减少拷贝。
- 存储类 I/O 还要关注队列、缓存、对齐、重试、超时和错误处理。

容易被追问：

- `read/write` 和 `ioctl` 区别。
- 阻塞和非阻塞区别。
- MMIO 和普通内存访问区别。
- I/O 出错后如何定位：日志、返回值、超时、设备状态、内核堆栈。

## 2. 线程同步怎么回答

面试答法：

> 线程同步的核心是保护共享资源，避免竞态条件。常用手段有互斥锁、信号量、条件变量、读写锁、原子操作。在 Linux 内核里还要区分上下文：互斥锁可能睡眠，适合进程上下文；自旋锁不会睡眠，适合短临界区或中断相关场景；如果只是简单计数，可以用原子操作。

常见选择：

| 场景 | 同步方式 |
| --- | --- |
| 普通共享数据保护 | mutex |
| 中断上下文或持锁时间很短 | spinlock |
| 资源计数或生产者消费者 | semaphore |
| 读多写少 | rwlock |
| 简单计数 | atomic |

重点坑：

- `volatile` 不是线程同步手段，不能保证原子性。
- 持有自旋锁时不能睡眠。
- 中断上下文不能使用会睡眠的锁。
- 临界区越短越好。

## 3. 驱动的 class 注册架构

面试答法：

> Linux 设备驱动可以从设备模型理解：bus、device、driver、class。以字符设备为例，一般先申请设备号，再初始化并注册 `cdev`，然后通过 `class_create` 和 `device_create` 在 sysfs 和 `/dev` 下创建用户可见节点，最后用户态通过 `open/read/write/ioctl` 调到驱动的 `file_operations`。

典型流程：

```c
alloc_chrdev_region(&devt, 0, 1, "demo");
cdev_init(&demo_cdev, &demo_fops);
cdev_add(&demo_cdev, devt, 1);

demo_class = class_create("demo_class");
device_create(demo_class, NULL, devt, NULL, "demo");
```

退出流程反过来：

```c
device_destroy(demo_class, devt);
class_destroy(demo_class);
cdev_del(&demo_cdev);
unregister_chrdev_region(devt, 1);
```

回答时可以强调：

- `class_create/device_create` 主要是为了让用户态看到设备节点。
- 真正的操作入口是 `file_operations`。
- 注册和注销顺序要匹配，否则容易资源泄漏。

## 4. 两个无序链表如何排成一个有序链表

面试答法：

> 如果两个链表本身都是无序的，不能直接用归并。比较稳的做法是先把两个链表连接起来，然后对整体链表做排序。链表适合用归并排序，因为不需要随机访问，时间复杂度是 `O(n log n)`，额外空间可以做到比较小。数据量小的时候也可以用插入排序，但最坏是 `O(n^2)`。

步骤：

1. 将两个链表首尾连接成一个链表。
2. 用快慢指针找到中点。
3. 递归拆分左右链表。
4. 分别排序后合并两个有序链表。

如果面试官说“两个链表已经有序”，那答案变成：

> 用双指针归并，谁小取谁，时间复杂度 `O(m+n)`。

## 5. `volatile`、`static`、指针大小

### `volatile`

面试答法：

> `volatile` 是告诉编译器不要优化这个变量的访问，每次都要从内存或寄存器位置重新读取。常见场景是硬件寄存器、中断服务程序和主流程共享的标志位。但 `volatile` 不保证原子性，也不能替代锁。

### `static`

面试答法：

> `static` 有两个常见作用。修饰全局变量或函数时，限制符号只在当前文件可见；修饰函数内部局部变量时，变量生命周期变成整个程序运行期间，但作用域仍在函数内。

### 指针大小

面试答法：

> 指针大小和平台位数有关，32 位系统通常是 4 字节，64 位系统通常是 8 字节。`sizeof(pointer)` 得到的是指针变量本身大小，不是它指向内存的大小。

## 6. 如何判断一个浮点数是否等于 1

面试答法：

> 一般不要直接用 `==` 判断浮点数，因为浮点数有精度误差。常用方式是设置一个误差范围 epsilon，判断差值的绝对值是否足够小。

```c
#include <math.h>

if (fabs(x - 1.0) < 1e-6) {
    /* x 可以认为等于 1 */
}
```

补充：

- `epsilon` 的大小要根据业务精度决定。
- 如果这个浮点数来自整数赋值或某些确定转换，可能可以精确表示，但面试里默认回答误差判断更稳。

## 7. 全局变量和局部变量同名时调用哪个

面试答法：

> 在 C 语言里，局部变量会遮蔽同名全局变量。在局部作用域内访问这个名字，优先访问局部变量。

```c
int value = 10;

void test(void)
{
    int value = 20;
    printf("%d\n", value);  /* 打印 20 */
}
```

注意：

- C 语言没有 C++ 的 `::value` 这种全局作用域访问方式。
- 实际编码中应避免同名，容易降低可读性。

## 8. 用两个栈实现一个队列

面试答法：

> 用两个栈，一个作为输入栈 `in`，一个作为输出栈 `out`。入队时直接压入 `in`；出队时如果 `out` 不为空，直接弹出 `out`；如果 `out` 为空，就把 `in` 中所有元素依次弹出并压入 `out`，这样顺序就反过来了，再弹出 `out`。

流程：

```text
enqueue(x):
    push x to in

dequeue():
    if out is empty:
        while in is not empty:
            push pop(in) to out
    return pop(out)
```

复杂度：

- 单次最坏 `O(n)`。
- 均摊复杂度 `O(1)`，因为每个元素最多进出两个栈各一次。

## 9. 如何通过堆栈抛出的地址定位到函数

用户态程序：

```bash
addr2line -e ./a.out 0x地址
gdb ./a.out
(gdb) info line *0x地址
```

如果程序开启 PIE 或 ASLR，需要先从 `/proc/<pid>/maps` 找到模块加载基地址，然后用：

```text
真实偏移 = 报错地址 - 模块加载基地址
```

再用 `addr2line` 查。

内核态：

```bash
cat /proc/kallsyms | grep 函数名
addr2line -e vmlinux 0x地址
nm module.ko | grep 函数名
```

如果是 `.ko` 模块中的地址，需要结合模块加载地址、模块符号表和对应版本的 `.ko` 文件定位。

面试答法：

> 关键是要有带符号信息的可执行文件或 `vmlinux/.ko`，先确认地址属于哪个模块，再扣掉加载基地址，最后用 `addr2line`、`gdb`、`nm` 或 `objdump` 映射到函数和源码行。

## 10. ARM 中配置 DSDT 怎么回答

先纠正概念：

> DSDT 是 ACPI 里的表，全称 Differentiated System Description Table，用 ASL 描述设备、资源、中断、电源管理方法等，再编译成 AML 给固件加载。传统嵌入式 ARM 更常用 Device Tree；ARM 服务器或 UEFI/ACPI 平台可能会使用 ACPI/DSDT。

面试答法：

> 如果是普通嵌入式 ARM 平台，我会优先想到设备树 DTB，通过 DTS 描述外设寄存器、中断、时钟、GPIO 等资源；如果是 ARM 服务器或支持 ACPI 的平台，DSDT 通常由 BIOS/UEFI 固件提供，开发时修改 ASL 文件，再用 `iasl` 编译成 AML，随固件一起加载，Linux 启动时通过 ACPI 解析设备资源。

需要记住：

- DTS/DTB：嵌入式 ARM 常见。
- ACPI/DSDT：x86 常见，ARM 服务器也可能用。
- DSDT 不是普通驱动里随便配置的文件，更多在固件/BIOS/UEFI 层。

## 面试回答模板

以后遇到技术问题，可以按这个顺序答：

1. 这个概念是什么。
2. 一般在哪些场景用。
3. 有什么坑或限制。
4. 我在项目里怎么遇到或怎么定位。

例子：

> `volatile` 是告诉编译器不要优化变量访问，常用于硬件寄存器或中断共享标志位。但它不能保证线程安全，所以如果涉及多个线程同时读写，还是要用锁或原子操作。我之前做驱动调试时，对硬件状态寄存器这类变量会关注是否需要按硬件访问语义读取。
